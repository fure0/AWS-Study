## ElasticCache도 달았고요. 이정도면 충분할꺼 같은데요. 그렇지 않습니다.
- 아무리 캐쉬를 달고, Read Replica를 붙이고 해도 한계는 있음
- 수십만 수백만명이 접근하게 되면 어떨까?
- 더 미래를 대비해 데이터베이스를 분산하는 작업을 해야 할거 같음

## DynamoDB는 거의 무한 성능을 사용 할 수 있다고 합니다.
- RDS의 가장큰 문제는 과도한 읽기 쓰기에 취약하다는 것
- 그리고 성능이 부족하면 인스턴스를 여러개 만들어서 성능을 확장하는 Scale-out 방식이 아닌 인스턴스의 타입을 변경하여 성능을 올리는 Scale-up 방식을 사용해야 함
- Read Replica로 읽기 전용이지만 Scale-out이 되긴 하지만 한계가 있음
- 관계형 데이터베이스는 이런 대규모 데이터를 다루는 샤딩(Sharding)이 있지만 너무 손이 많이 감

## DynamoDB는 NoSQL중에 Key/Value 형태로 데이터를 저장하는 데이터 베이스 입니다.
- 무하한 성능에, 관리도 필요없는 key/value 데이터 베이스
- 사용한 만큼만 비용을 내면 됨

## 이미 모든 데이터를 RDS에 저장하고 있어서 아무리 DynamoDB가 고성능이라지만 쉽게 데이터를 옳길수가 없습니다. 독립적인 테이블만 일단 옮기도록 하죠.
- 모든 데이터를 RDS에서만 관리 하는게 아니고 다른 DB에 저장해서 사용하는 것도 괜찮은거 같음
- 한마디로 데이터베이스를 여러개 사용하는 것임. 이렇게 RDS의 부하를 분산하도록 하자
- 관계형 데이터베이스는 관계로 이루어져 있기 때문에 아무 테이블의 데이터를 빼서 옮기면 안됨
- 독립적인 테이블만 뺴서 DynamoDB로 일단 옮기자

## 이제 DB도 뷴산 했으니, 걱정 없을거 같네요. 사용자가 얼마든지 늘어 나도 충분히 감당 할 수 있을거 같아요.
- DynamoDB에 성능을 더 높이려면 DynamoDB Accelerator(DAX)를 붙이면 DynamoDB의 캐시 기능을 한다고 함
- 중요한건 캐시 단다고 항상 빨라지고 비용이 줄고 하는건 아님
- 한번 불러와서 계속 사용하는 데이터가 많을때 유리함
